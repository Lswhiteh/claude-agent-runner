claude-agent-runner

Autonomous Claude agents that turn Linear issues into GitHub PRs

A ~2,500-line bash orchestrator with hooks, skills, tracing, and multi-agent coordination.

  • Polls Linear on a cron schedule (e.g., every 5 minutes)
  • Each issue gets an isolated git worktree + optional Docker DB
  • Claude runs autonomously with guardrail hooks
  • CI gate validates; retry loop auto-fixes failures
  • Creates draft PR, posts report to Linear, notifies Slack

Configuration

// ~/.config/claude-agents/config.json
{
  "label": "Agent",          // Linear label that triggers pickup
  "max_parallel": 3,           // concurrent agents per team
  "max_ci_retries": 3,         // CI fail → fix → retry attempts
  "orchestrator": {
    "enabled": true,
    "label": "Orchestrate",   // triggers multi-agent decomposition
    "max_subtasks": 8
  },
  "workspaces": {
    "mycompany": {
      "api_key_env": "LINEAR_API_KEY_MYCO",
      "teams": {
        "Engineering": { "repo": "/path/to/repo", "local_db": "postgres" }
      }
    }
  }
}

Workspaces = Linear orgs. Each team maps to a local git repo.

The Polling Loop

# Cron fires every 5 minutes
*/5 * * * * source ~/.config/claude-agents/secrets.env && claude-agent-runner
1
Acquire per-repo lock (PID-based) to prevent overlapping runs
2
Clean up worktrees for merged/closed PRs
3
Count available slots: max_parallel − active agents (by PID)
4
Fetch labeled issues via Linear GraphQL (up to available slots)
5
Spawn a background subprocess for each issue

The runner is stateless — all state lives in worktrees, Linear, GitHub, and trace files.

Issue Lifecycle

What happens when a new issue is picked up

1. Worktree & environment

  • git worktree add from origin/main
  • Branch: agent/ENG-123-title-slug
  • Copies .env files from repo root
  • Optionally starts Docker DB container

2. Linear update

  • State → In Progress
  • Remove Agent label (prevents re-pickup)

3. Claude invocation

export CLAUDE_AGENT_MODE=1
export CLAUDE_AGENT_ISSUE_ID="ENG-123"

claude --permission-mode bypassPermissions \
  -p "You are working on ENG-123..." \
  --max-turns 500

Claude reads CLAUDE.md, explores the codebase, writes tests, implements, commits.

Auto-Review & Auto-Fix

Auto-Review

A read-only Claude instance (--permission-mode plan) reviews the diff and outputs structured JSON:

{
  "verdict": "request_changes",
  "blocking": ["Missing input validation"],
  "important": ["Add aria-label"],
  "nits": [...],
  "praise": [...]
}

Posted via gh pr review with the matching verdict.

Auto-Fix Loop

If verdict = request_changes:

1
Gather all PR feedback
2
Spawn Claude to address feedback
3
Run CI, push, re-review
4
Repeat until approved or max iterations
claude-agent-runner --auto-review    # review only
claude-agent-runner --auto-fix       # review + fix loop

Runner side

Detects the blocked file and:

Orchestration: Multi-Agent

For issues spanning multiple files/concerns

Trigger: issue labeled Agent + Orchestrate

Phase 1 — Decompose

Claude analyzes the issue, writes subtask JSON files with file scopes, skills, and dependencies.

Phase 2 — Create Issues

Subtask JSONs become Linear child issues with dependency relations and agent labels.

Phase 3 — Supervise

Polls Linear every 30s. Workers are picked up by the normal polling loop as scoped agents.

Phase 4 — Integrate

Pull all changes, run CI on the integrated result. Fix agent if needed.

Phase 5 — Combined PR

Single PR with merged reports from all subtasks. Posted to Linear + Slack.

Scoped Workers

Subtask agents are sandboxed to their assigned files via the scope-guard hook.

// subtask-001.json
{
  "id": "ST-1",
  "title": "Add auth middleware",
  "skill": "backend",
  "scope": [
    "src/middleware/auth.ts",
    "src/lib/jwt.ts"
  ],
  "off_limits": ["src/db/*"],
  "shared_files": ["src/types.ts"],
  "depends_on": []
}

Enforcement

  • Write only to files in scope
  • Read anything in the repo
  • Shared files are read-only
  • If out-of-scope changes needed → write a scope-overflow document
export CLAUDE_AGENT_SCOPED=1
export CLAUDE_AGENT_SCOPE_FILE="subtask.json"

Summary

  • Stateless orchestrator — cron-friendly, all state external
  • Isolated worktrees — no conflicts between agents
  • 11 guardrail hooks — destructive-command blocking, scope enforcement, test requirements
  • Multi-agent orchestration — decompose, supervise, integrate
  • CI retry loop — self-healing on failures
  • Feedback resume — human-in-the-loop without restarting
  • Pipeline & spec — idea → spec → issues → PRs
  • Full observability — traces, logs, Slack, timeline viewer

~2,500 lines of bash. No framework. No build step.
